# **1\. Tư duy kiến trúc server**

Server của dự án nên dùng hướng Clean Architecture kết hợp Feature Folder. Nghĩa là code không chia theo từng người thành nhiều backend riêng, mà chia thành các layer chuẩn. Trong mỗi layer, code lại được tách theo feature nghiệp vụ như Auth, KYC, RoomingHouses, Rooms, Admin và Public.

| Câu chốt Một server, một solution, nhiều project theo layer. Mỗi người code trong feature được phân công, nhưng vẫn dùng chung Domain, Contracts, ApiResponse, ErrorCodes và AppDbContext. |
| :---- |

Cách này giúp team tránh các lỗi phổ biến: controller bị nhồi logic, DTO bị tạo trùng, enum bị viết bằng string lung tung, database configuration bị lẫn trong AppDbContext, hoặc mỗi thành viên tự trả response theo một format khác nhau.

·        Api chỉ nhận request và trả response.  
·        Application xử lý nghiệp vụ.  
·        Domain chứa entity, enum, constant.  
·        Infrastructure làm việc với database và service kỹ thuật.  
·        Contracts chứa request/response DTO, ApiResponse, ErrorCodes.

## **4.1. Controllers**

Controllers khai báo endpoint. Controller chỉ nên nhận request, gọi service trong Application, rồi trả ApiResponse. Không viết logic dài, không query DbContext trực tiếp nếu không thật sự cần.

| Controller | Phụ trách | Ví dụ endpoint |
| ----- | ----- | ----- |
| AuthController | Đăng ký, đăng nhập, OTP, Google login, refresh token, logout | POST /api/auth/login |
| MeController | Thông tin user hiện tại, roles, onboarding status | GET /api/me |
| ProfileController | Cập nhật hồ sơ cá nhân | PUT /api/me/profile |
| KycController | User gửi và xem trạng thái KYC | POST /api/kyc/submit |
| AdminKycController | Admin duyệt/từ chối KYC | POST /api/admin/kyc/{id}/approve |
| RoomingHousesController | Chủ trọ tạo/sửa/submit khu trọ | POST /api/rooming-houses |
| RoomsController | Chủ trọ tạo/sửa phòng, giá theo số người | POST /api/rooming-houses/{id}/rooms |
| AmenitiesController | Quản lý tiện nghi hoặc gán tiện nghi | GET /api/amenities |
| LeasePoliciesController | Chính sách thuê/cọc/gia hạn/discount | POST /api/rooming-houses/{id}/lease-policy |
| AdminRoomingHousesController | Admin duyệt/từ chối khu trọ | POST /api/admin/rooming-houses/{id}/approve |
| PublicRoomingHousesController | Người thuê xem danh sách/chi tiết khu trọ public | GET /api/public/rooming-houses |

 

## **4.2. Middlewares**

Middlewares xử lý logic chạy qua mọi request. File quan trọng nhất là ExceptionHandlingMiddleware.cs để bắt lỗi từ Application và trả về ApiErrorResponse chuẩn. Nhờ vậy frontend không phải xử lý nhiều kiểu lỗi khác nhau.

## **4.3. Extensions**

Extensions giúp Program.cs gọn hơn. Ví dụ AuthenticationExtensions.cs cấu hình JWT, SwaggerExtensions.cs cấu hình Swagger, CorsExtensions.cs cấu hình cho React gọi API.

## **4.4. appsettings**

appsettings.json chứa cấu hình như ConnectionStrings, JWT secret, Google OAuth, file storage. Không hard-code connection string hoặc secret trực tiếp trong service.

## **7.1. Persistence/AppDbContext.cs**

AppDbContext là class đại diện database trong code. Nó khai báo DbSet cho các entity như Users, KycVerifications, RoomingHouses, Rooms. AppDbContext không nên chứa quá nhiều cấu hình chi tiết; cấu hình từng bảng nên tách sang Configurations.

## **7.2. Persistence/Configurations**

Configurations chứa EF Core Fluent API cho từng entity. Mỗi entity nên có một file configuration riêng để dễ đọc, dễ review và tránh AppDbContext bị quá dài.

| Folder | File ví dụ | Cấu hình thường có |
| ----- | ----- | ----- |
| Configurations/Users | UserConfiguration.cs | Table name, email unique index, max length, enum conversion, relationship với profile/roles/tokens. |
| Configurations/Users | KycVerificationConfiguration.cs | Status enum string, citizen id hash, reviewed\_by\_admin\_id, relationship với user/admin. |
| Configurations/Properties | RoomingHouseConfiguration.cs | Landlord FK, approval status, visibility status, soft delete field, approve admin FK. |
| Configurations/Properties | RoomConfiguration.cs | RoomingHouse FK, room status, max occupants, area, relationship với price tiers/images. |
| Configurations/Properties | PropertyImageConfiguration.cs | Ảnh chỉ thuộc RoomingHouse hoặc Room, object key, image url, cover, sort order. |

 

## **7.3. Migrations**

Migrations là nơi EF Core tạo file thay đổi database. Team không nên sửa tay migration nếu không hiểu rõ. Khi entity/config thay đổi thì tạo migration mới, review kỹ rồi mới update database.

## **7.4. SeedData**

SeedData chứa dữ liệu mặc định như roles Admin/Tenant/Landlord và amenities mẫu. Seed giúp database mới có dữ liệu cơ bản để chạy thử và test API.

| ----- | ----- | ----- | ----- |

| Người 2 | Profile / Role / Me / Onboarding | Application/Users, Profiles, Roles; Contracts/Users, Profiles, Roles; Entity: UserProfile, Role, UserRole | MeController, ProfileController |

| Người 4 | Rooming House / Room / Price / Amenity / Image | Application/RoomingHouses, Rooms, Amenities, PropertyImages; Contracts tương ứng; Entities/Properties | RoomingHousesController, RoomsController, AmenitiesController |
| Người 5 | Admin House Approval / Public Listing / Lease Policy / Integration | Application/Admin, Public, LeasePolicies; Contracts/Admin, Public, LeasePolicies; Entity: LeasePolicy | AdminRoomingHousesController, PublicRoomingHousesController, LeasePoliciesController |

 

## **9.1. Rule tránh đụng nhau**

·        Người 4 không tự insert user\_roles để cấp Landlord. Việc grant role nên đi qua RoleService hoặc AdminRoomingHouseService.  
·        Người 3 không sửa logic tạo khu trọ hoặc phòng. KYC chỉ xử lý xác minh danh tính.  
·        Người 1 không tự sửa bảng rooming\_houses/rooms nếu không liên quan auth.  
·        Người 5 khi approve khu trọ phải kiểm tra điều kiện và gọi service cấp Landlord đúng flow.  
·        Mọi thay đổi enum/state phải có thông báo team vì frontend và backend đều phụ thuộc.

# **10\. Quy chuẩn gọi API và xử lý response/error**

Mọi API call phải đi qua service của feature. Component không gọi axios trực tiếp.

| Page/Component   ↓ gọi hook useLogin / useMyKyc / usePublicRoomingHouses   ↓ gọi service authApi.login / kycApi.getMyKyc / publicRoomingHouseApi.search   ↓ dùng shared axiosClient shared/api/axiosClient.ts   ↓ gọi backend ASP.NET Core Web API   ↓ trả ApiResponse\<T\> hoặc ApiErrorResponse Client render loading/success/error state |
| :---- |

 

| File | Nội dung nên có |
| :---- | :---- |
| axiosClient.ts | Base URL, timeout, Authorization header, refresh token interceptor nếu triển khai. |

| apiResponse.types.ts | ApiResponse\<T\>, ApiErrorResponse, PagedResult\<T\>. |

| // features/auth/services/authApi.ts export const authApi \= {   login: (payload: LoginRequest) \=\>     axiosClient.post\<ApiResponse\<LoginResponse\>\>(ENDPOINTS.AUTH.LOGIN, payload),     logout: () \=\>     axiosClient.post\<ApiResponse\<void\>\>(ENDPOINTS.AUTH.LOGOUT), }; |
| :---- |

 

Ví dụ component không nên làm:

| // Không nên axios.post('http://localhost:5000/api/auth/login', formValues); |
| :---- |

 

| ErrorCode Frontend nên hiển thị lỗi theo errorCode backend trả về, ví dụ KYC\_REQUIRED, HOUSE\_INVALID\_STATUS, ROOM\_NOT\_FOUND. Không nên chỉ phụ thuộc vào message text vì message có thể đổi. |
| :---- |

# **14\. Naming convention và import rule**

| Loại file | Quy ước đặt tên | Ví dụ |
| :---- | :---- | :---- |
| Page component | PascalCase \+ Page.tsx | LoginPage.tsx, MyKycPage.tsx |
| UI component | PascalCase.tsx | RoomCard.tsx, KycStatusBadge.tsx |
| Service API | camelCase \+ Api.ts | authApi.ts, kycApi.ts |
| Hook | use \+ PascalCase.ts | useLogin.ts, usePublicRoomingHouses.ts |
| Type file | feature.types.ts | auth.types.ts, room.types.ts |
| Schema file | feature.schema.ts | profile.schema.ts |
| Constants file | feature.constants.ts | kyc.constants.ts |
| Route path | UPPER\_SNAKE\_CASE hoặc object constant | ROUTE\_PATHS.AUTH.LOGIN |

 

Import rule nên dùng:

·        Feature có thể import từ shared.

·        Feature không nên import sâu từ feature khác, trừ khi qua index.ts đã thống nhất.

·        shared không được import từ features để tránh vòng phụ thuộc.

·        Không viết endpoint string trực tiếp trong component.

·        Không viết enum/status string tự do trong UI; dùng constants/types đã thống nhất.

# **Kết luận**

Cấu trúc client này giúp dự án React bám đúng module nghiệp vụ của server, dễ chia cho 5 người làm song song, dễ review, dễ mở rộng sang các interval sau. Điểm quan trọng nhất là không để logic nghiệp vụ và API bị rải rác trong component; mọi feature phải có service, type, hook và UI state rõ ràng.

| Chốt chuẩn | Nội dung |
| :---- | :---- |
| Client | React \+ TypeScript theo feature/module. |
| Feature chính | auth, me, profile, kyc, landlord-property, admin, public-listing. |
| Shared | api, components, hooks, types, constants, utils. |
| Guard | ProtectedRoute, RoleGuard, OnboardingGuard. |
| Rule | Client hỗ trợ UX, backend quyết định nghiệp vụ cuối cùng. |
